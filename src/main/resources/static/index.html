Processing Pipeline

Step 1
Opt-out Check
Single strong read from consumer_preferences. Loads opted_out and timezone together — both reused downstream. No eventual consistency.
→ 1 DB read · serves Steps 1 & 3
KEY DESIGN
Step 2
3-Day Suppression
Do not contact a consumer if already reached in the last 3 days (any channel).
SELECT EXISTS (... status IN ('SENT','DELIVERED') AND sent_at > :since)
↳ partial composite index → O(log n) at 1M+ rows
Step 3
FDCPA Window
Contact only 08:00–18:00 in the consumer's local timezone. HELD if outside window. Timezone was loaded in Step 1 — no extra DB call here.
→ in-memory timezone · pure time arithmetic
Steps 4–5
Render & Send
Template render → mini-Miranda validation (FDCPA) → simulated Twilio / SendGrid send.
→ transactional outbox
⚡
Suppression index design: CREATE INDEX idx_notifications_recent_contact ON notifications (tenant_id, consumer_id, sent_at DESC) WHERE status IN ('SENT', 'DELIVERED')
Partial index — only indexes rows that matter. Composite order matches the query exactly: tenant → consumer → recency. Postgres satisfies the EXISTS with an index-only scan.

updates every 30s · consumers are seeded across all 5 zones
Notifications sent to consumers in a CLOSED timezone are held until 08:00 local time next day → reflected in FDCPA Hold count in load test results.
